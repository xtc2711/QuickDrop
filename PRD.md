## 1. 需求背景

### 1.1 当前状况

跨设备文件传输是每个拥有多台设备的用户几乎每天都会遇到的场景。当前市场上的解决方案存在明显的共性痛点：

- **微信/QQ 等即时通讯工具**: 超过 1GB 的文件无法发送，视频和图片被强制压缩，画质严重受损。同一局域网内的传输也需要经过云端服务器中转，速度受限于公网带宽。`[PM hypothesis]`
- **AirDrop (Apple 生态)**: 体验极佳、速度极快，但仅限 Apple 设备之间。Windows 和 Android 用户被完全排除在外。`[Expert judgment]`
- **USB 数据线**: 可靠但麻烦——需要数据线、驱动安装、文件管理操作繁琐。`[PM hypothesis]`
- **网盘/U 盘/邮件附件**: 流程冗长——上传到云端再下载，速度慢且受文件大小限制。`[Research-backed]`
- **开源 P2P 工具 (如 Snapdrop 等)**: 用户体验粗糙，浏览器兼容性问题多，缺乏账户体系和设备管理能力。`[Expert judgment]`

### 1.2 核心问题

| 问题 | 严重程度 | 受影响用户范围 |
|---|---|---|
| 视频/音频被强制压缩，画质损失不可逆 | 高 | 所有通过 IM 工具传输媒体的用户 |
| 大文件（>1GB）传输无可靠通道 | 高 | 视频创作者、设计师、开发者 |
| 跨平台（Windows ↔ Mac ↔ Android ↔ iOS）传输困难 | 高 | 同时使用不同操作系统设备的用户 |
| 传输需要经过云端中转，速度慢且有隐私顾虑 | 中 | 对数据隐私敏感的用户 |
| 每次传输需要手动配对，重复操作成本高 | 中 | 频繁在固定设备间传输的用户 |

### 1.3 不解决这些问题的后果

- 用户在设备间传输文件时持续面临摩擦，生产力受损
- 已有的高质量媒体内容（4K 视频、无损音频、RAW 照片）在传输过程中质量降级
- 用户被迫在多个工具之间切换（AirDrop 传 Mac→iPhone、微信传电脑→安卓、USB传大文件），体验割裂
- 对产品失去信任，转向更新的竞品

## 2. 产品目标

### 2.1 产品愿景

让任意两台设备之间的文件传输像 AirDrop 一样简单，但覆盖 Windows、macOS、Android、iOS 所有平台。

### 2.2 核心目标

| 目标 | 衡量标准 | 优先级 |
|---|---|---|
| **无损传输**: 任何格式的文件在传输过程中不被压缩或修改 | 传输前后的文件 SHA256 完全一致 | P0 |
| **零配置自动配对**: 同一账户登录的设备自动发现并建立连接，无需手动操作 | 登录后 5 秒内设备出现在列表中 | P0 |
| **智能通道选择**: 自动选择最优传输通道（蓝牙/热点 → 局域网 → 中继） | 局域网场景下延迟 < 50ms 建立 P2P 连接 | P0 |
| **跨平台全覆盖**: 支持 Windows、macOS、Android、iOS | 四大平台均可用 | P0 |
| **多设备并发**: 同一账户可在多台设备同时在线 | 不限并发设备数 | P0 |
| **大文件支持**: 无文件大小上限（受设备存储和网络带宽限制） | 可传输 10GB+ 文件 | P1 |

### 2.3 非目标（明确不做）

- 不做文件云端存储或同步（不是网盘）
- 不做文件格式转换或预览（不替代播放器/查看器）
- 不做群组传输或广播（当前仅一对一传输）
- 不做端到端加密的额外实现（WebRTC 内置 DTLS 加密已满足需求；未来版本可扩展）

## 3. 产品定位与竞品分析

### 3.1 产品定位

QuickDrop 定位为 **"跨平台的 AirDrop"**——一个覆盖所有主流操作系统的点对点文件传输工具，强调无损、快速、零配置。

目标用户心智模型: "要在电脑和手机之间传个文件 → 打开 QuickDrop → 拖进去 → 完成。"

### 3.2 竞品对比

| 维度 | QuickDrop | AirDrop | Snapdrop | Send Anywhere | 微信文件传输助手 |
|---|---|---|---|---|---|
| **跨平台** | Win + Mac + Android + iOS | 仅 Apple | Web (所有平台浏览器) | 全平台 | 全平台 |
| **传输方式** | P2P 直连 (蓝牙/局域网/中继) | P2P (蓝牙+WiFi) | P2P (WebRTC 局域网) | P2P + 中继 | 服务器中转 |
| **无损传输** | 是 | 是 | 是 | 是 | 否（图片/视频压缩） |
| **账户系统** | 有，自动配对 | Apple ID 自动 | 无 | 无，取件码 | 微信账号 |
| **配对方式** | 自动配对 + 扫码 + 配对码 | 自动发现 | 网页配对 | 6位码 | 扫码 |
| **文件大小限制** | 不限 | 不限 | 浏览器限制 | 有限 | <1GB |
| **离线可用** | 局域网直连无需公网 | 是 | 是 | 否 | 否 |
| **原生应用** | 是 (Tauri) | 是 | 否 (Web) | 是 | 是 |
| **数据传输路径** | 仅在设备间 | 仅在设备间 | 仅在设备间 | 中继服务器 | 腾讯服务器 |

**分析结论**: AirDrop 体验最佳但生态封闭；Snapdrop 跨平台但缺账户和原生能力；Send Anywhere 提供类似功能但有文件大小限制且依赖中继。QuickDrop 的核心差异化在于 **"账户自动配对 + 多通道智能切换 + 完全无损"** 的组合。`[Expert judgment]`

### 3.3 差异化优势

1. **同账户自动配对**: 登录后不需要任何手动操作，设备自动出现在列表中。这是与 Snapdrop、Send Anywhere 等工具的关键区别
2. **三层通道自动降级**: 蓝牙→局域网→中继，确保任何网络环境都能传输
3. **对媒体的严格无损承诺**: 视频、音频保持原始编码和封装，不做任何转码或重压缩

## 4. 目标用户与使用场景

### 4.1 用户画像

| 画像 | 典型身份 | 技术熟练度 | 痛点 | 频率 |
|---|---|---|---|---|
| **多设备办公族** | 设计师、开发者、产品经理 | 中高 | 在 Mac 和 iPhone 之间频繁传文件；工作电脑（Win）和个人设备（Android/iOS）之间传输困难 | 每天 3-10 次 |
| **内容创作者** | 摄影师、视频剪辑师 | 中高 | 从相机/手机传输大体积 RAW 照片或 4K 视频到电脑编辑，对画质要求极高 | 每次拍摄后 10-50 次 |
| **跨平台用户** | 同时使用 Windows 和 Mac、Android 和 iOS 的用户 | 中 | 需要在不同生态的设备间偶尔传文件，现有方案体验差 | 每周 2-5 次 |
| **普通家庭用户** | 给家人传照片、视频 | 低 | 只会用微信传输，但画质被压缩严重；不会配置复杂的网络设置 | 每周 1-3 次 |

### 4.2 核心使用场景

**场景 1: 办公室跨设备传输（P0）**

> 用户 Alice 是设计师。她在 MacBook Pro 上完成了一版设计稿（.fig，约 200MB），需要发给客户的 iPhone。她打开 QuickDrop（已登录），设备列表中自动显示同账户的 iPhone。她把设计稿拖拽到 iPhone 设备上，10 秒后传输完成。客户直接在手机上查看。

**场景 2: 拍摄现场大文件回传（P0）**

> 摄影师 Bob 用相机拍摄了约 20GB 的 4K 视频素材。他通过读卡器将素材拷贝到 Windows 笔记本上，然后打开 QuickDrop。他的 Android 手机已登录同一账户，自动显示在设备列表中。他选中所有素材拖拽到手机设备图标上，通过 WiFi 6 局域网以约 80MB/s 的速度传输。所有视频保持原始编码，无任何压缩。

**场景 3: 无网络环境下的应急传输（P1）**

> Carol 和她的同事在客户的会议室开会，现场没有 WiFi 可用。Carol 需要在她的 Windows 电脑和同事的 MacBook 之间传一个 PPT 文件。她打开 QuickDrop，点击"添加设备"生成一个 6 位配对码。同事扫码后连接，两台电脑通过蓝牙直连。文件传输完成后，连接自动断开。

**场景 4: 居家跨生态传输（P1）**

> David 用 Android 手机拍摄了孩子的视频（约 3GB），想传给 MacBook 上剪辑。两部设备都连接了家中 WiFi。他打开 QuickDrop 后，两台设备自动配对。他选择视频文件，传输通过局域网 P2P 直连，全程约 30 秒完成。视频画质与手机原始拍摄完全一致。

### 4.3 使用前提

- 设备需要安装 QuickDrop 客户端（桌面端或移动端）
- 自动配对功能需要用户登录账户；未登录用户可使用扫码/配对码
- P2P 直连不需要互联网连接（局域网内即可）
- 中继传输需要两端都能访问公网

```
│   │  └────────────────────────┘   │    │
│   │  ┌────────────────────────┐   │    │
│   │  │ 密码                    │   │    │
│   │  │ ....................   │   │    │
│   │  └────────────────────────┘   │    │
│   │                              │    │

│   │  └────────────────────────┘   │    │
│   │                              │    │
│   │  没有账户？[注册]             │    │
│   └──────────────────────────────┘    │
└──────────────────────────────────────┘
```

#### 功能描述

**F1 - 用户注册**

| 维度 | 描述 |
|---|---|
| **业务逻辑** | 用户输入邮箱、密码，系统校验格式后创建账户，使用 bcrypt 加密存储密码。创建成功后自动生成设备记录，签发 JWT Access Token 和 Refresh Token。 |
| **交互逻辑** | 用户点击"注册"→ 切换到注册表单 → 填写邮箱和密码 → 点击提交按钮 → 按钮变为加载状态 → 注册成功后自动跳转到主界面 → 注册失败时显示明确错误提示。 |
| **规则约束** | 邮箱: 文本，最大 255 字符，符合邮箱格式正则，必填，唯一性校验。密码: 最少 8 位，必须包含大写字母、小写字母、数字，最多 128 字符，必填。设备名称: 由客户端自动检测并上报（如 "MacBook Pro"），用户可在设置中修改。 |

**F2 - 用户登录**

| 维度 | 描述 |
|---|---|
| **业务逻辑** | 验证邮箱和密码匹配，生成新的 Access Token 和 Refresh Token。记录当前设备的登录会话到服务端。如果设备已存在记录则更新 last_seen；如果设备超过`[待确认]`天未登录则标记为不活跃。 |
| **交互逻辑** | 用户输入邮箱和密码 → 点击登录 → 按钮变为加载状态 → 成功后跳转到主界面，设备列表自动加载 → 登录失败（密码错误/账户被禁用）显示明确错误。 |
| **规则约束** | 连续 5 次登录失败后，该账户锁定 15 分钟。 |

### 5.3 模块 2: 设备自动配对

#### 原型: 自动配对时序

```
 设备A (已登录, 在线)          信令服务             设备B (新登录)
      │                         │                      │
      │  WebSocket 已连接        │                      │
      │  (携带 JWT Token)       │                      │
      │────────────────────────►│                      │
      │                         │                      │
      │                         │   设备B登录并连接     │
      │                         │◄─────────────────────│
      │                         │                      │
      │                         │  验证 Token，发现     │
      │                         │  同 userId           │
      │                         │                      │
      │  same_account_device_    │                      │
      │  online { "iPhone" }    │                      │
      │◄────────────────────────│                      │
      │                         │ same_account_device_ │
      │                         │ online { "MacBook" } │
      │                         │─────────────────────►│
      │                         │                      │
      │  ── 客户端自动发起 ──    │                      │
      │  WebRTC ICE 连接尝试     │                      │
      │═════════════════════════╪══════════════════════►│
      │          DataChannel 就绪                       │
```

#### 功能描述

**F3 - 自动配对**

| 维度 | 描述 |
|---|---|
| **业务逻辑** | 用户登录后，WebSocket 连接携带 JWT Token。服务端解析 Token 获取 userId，将当前设备注册到该用户的在线设备列表。立即查询同 userId 的其他在线设备，向双方推送设备上线通知。客户端收到通知后自动发起 WebRTC ICE 连接建立尝试。ICE 连接建立后，设备出现在对方的设备列表中，状态为"已配对"。 |
| **交互逻辑** | 用户登录成功 → 主界面加载 → 设备列表显示同账户其他在线设备 → 如果设备尚未建立 P2P 连接，显示"连接中"状态 → 连接建立后变为"已连接" → 如有新设备上线，自动触发通知提示。整个过程无需用户操作。 |
| **规则约束** | 自动配对仅在同账户设备间生效。设备离线后取消在线标记，30 秒无心跳视为离线。每个 P2P 连接有 30 秒超时，超时后降级为中继或标记为不可用。 |

**F4 - 扫码配对**

| 维度 | 描述 |
|---|---|
| **业务逻辑** | 桌面端点击"添加设备"→ 选择"二维码"→ 服务端生成包含连接信息的二维码（含 WebSocket 地址和一次性配对令牌）。移动端扫码后解析信息，连接到信令服务完成配对。配对成功后，双方建立 WebRTC 连接。 |
| **交互逻辑** | 桌面端: 添加设备 → 显示二维码弹窗 + 6位配对码 → 等待扫描。移动端: 扫码 → 自动连接。配对完成后弹窗关闭，新设备出现在列表中。 |
| **规则约束** | 二维码有效期 2 分钟，超时后需要重新生成。扫码配对不要求双方登录同一账户。 |

**F5 - 配对码连接**

| 维度 | 描述 |
|---|---|
| **业务逻辑** | 桌面端点击"添加设备"→ 服务端生成 6 位数字配对码，关联到一个临时房间。移动端输入配对码后发送到信令服务，服务端验证后将双方加入同一房间，完成配对。 |
| **交互逻辑** | 桌面端: 显示 6 位数字（每位一个格子）。移动端: 切换到"输入配对码"Tab → 逐个输入数字（自动跳格）→ 6 位输完后自动或点击按钮发起连接。支持粘贴完整 6 位数字。 |
| **规则约束** | 配对码有效期 2 分钟。排除易混淆组合（如 123456、000000）。同一 IP 每 60 秒最多 5 次配对尝试。 |

## 6. 技术架构概述

> **说明**: 本节提供架构层面的总体设计，具体的数据库、API 和部署细节参见关联的技术设计文档 [AUTH_DESIGN.md](./AUTH_DESIGN.md) 和 [README.md](./README.md)。

### 6.1 分层架构

```
客户端层 (Windows / macOS / Android / iOS)
      │
      ├── WebSocket ──── 信令服务 (配对、设备发现、信令转发)
      │
      ├── HTTPS ──────── 认证服务 (注册、登录、Token 管理)
      │
      └── WebRTC P2P ── 文件数据直连（不经过任何服务器）

中继层: STUN/TURN 服务器（仅在无法直连时启用）
```

### 6.2 核心模块

| 模块 | 职责 |
|---|---|
| **认证服务** | 处理用户注册、登录、Token 签发与刷新、退出登录、设备管理。对外暴露 RESTful API。 |
| **信令服务** | 负责 WebSocket 连接管理、设备上线/下线广播、配对码管理、房间管理、WebRTC 信令透传。同时承担同账户设备自动配对的逻辑。 |
| **P2P 传输模块** | 客户端集成 WebRTC 引擎，负责 ICE 候选收集、DataChannel 建立、文件分块传输、完整性校验。 |

### 6.3 连接优先级策略

设备间的数据传输通道按以下顺序自动选择:

1. **蓝牙/热点直连**: 适用于短距离场景，通过蓝牙 LE 或 WiFi Direct 建立连接。传输速率约 3~10MB/s。
2. **局域网 P2P**: 同一 WiFi 网络下的高速直连，通过 ICE 框架自动发现局域网候选地址。传输速率可达 50~100MB/s+。
3. **TURN 中继**: ICE 无法建立直连时自动回退到 TURN 中继传输，数据通过中继服务器转发。传输速率取决于中继带宽。

通道选择完全由客户端 ICE 框架自动完成，用户无需干预。当前生效的通道类型在界面上以标签形式展示。

| A12 | 退出登录 | 退出后设备从同账户设备列表中移除，其他设备收到下线通知 |

### 9.2 性能验收

| 编号 | 验收项 | 通过条件 |
|---|---|---|
| P1 | 局域网传输速度 (WiFi 5) | 同一 5GHz WiFi 下传输 500MB 文件，平均速度 > 50MB/s |
| P2 | 自动配对延迟 | 从设备登录到出现在对方列表的时间，P50 < 5 秒，P95 < 10 秒 |
| P3 | 客户端内存占用 | 不传输文件时内存占用 < 100MB；传输 10GB 文件时峰值内存 < 300MB |
| P4 | 服务端并发 | 信令服务同时支持 1000 个 WebSocket 连接，CPU 使用率 < 50% |

### 9.3 安全验收

| 编号 | 验收项 | 通过条件 |
|---|---|---|
| S1 | 密码存储 | 用户密码以 bcrypt (cost=12) 哈希存储，不存明文 |
| S2 | Token 传输 | Access Token 和 Refresh Token 仅通过 HTTPS 传输 |
| S3 | Token 撤销 | 退出登录后，该 Token 在 1 秒内失效 |
| S4 | 文件数据安全 | 文件通过 DataChannel 的 DTLS 加密传输，抓包工具无法还原文件内容 |
| S5 | 速率限制 | 登录接口单 IP 每分钟超过 10 次请求后返回 429 |
